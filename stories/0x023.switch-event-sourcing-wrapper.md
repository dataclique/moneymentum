# Replace Ingestion Event Sourcing With Run Ledger

As an operator, I want each ingest attempt to be recorded as an explicit run, so
that crashes and manual cancellations remain visible without resetting the
database.

## Context

Ingestion status used to be a singleton CQRS aggregate. If the backend crashed
after recording `Started` but before recording `Completed` or `Failed`, the
database kept reporting `Running` forever. That forced unsafe database resets
before another ingest could start.

Ingestion now uses a plain `ingestion_runs` table. Each run has an ID, status,
timestamps, heartbeat, and optional failure reason. Apalis remains the durable
job queue, and each queued job carries the run ID it is responsible for.

## Acceptance Criteria

- [x] Scheduled ingestion runs are enqueued by built-in apalis-cron workers
      before the job executes.
- [x] `IngestionJob` carries a typed run ID and updates that run to `completed`
      or `failed`.
- [x] Backend startup marks leftover `running` rows as `failed`, making crashed
      runs visible and allowing subsequent ingests to start.
- [x] The old `cqrs-es`/`sqlite-es` dependencies and
      `src/wire.rs`/`src/lifecycle.rs` wrapper code are removed.
- [x] A forward-only SQL migration creates the `ingestion_runs` table and
      prevents more than one active `running` run.
- [x] README, ROADMAP, SPEC, config comments, and agent rules describe the new
      SQLite ingestion run ledger instead of CQRS.
