# Switch Event Sourcing To A Better Wrapper

As a developer, I want the backend to use `event-sorcery` instead of the current
`cqrs-es` plus github-referenced `sqlite-es` setup, so that aggregate IDs are
strongly typed, wiring mistakes are caught at compile time without the
hand-rolled phantom-type machinery in `src/wire.rs`, and the persistence backend
lives in one cohesive crate we control.

## Context

The backend currently uses `cqrs-es` 0.4.11 (event-sourcing framework) plus a
github-referenced `sqlite-es` fork (sqlite event store), wrapped by a homemade
type-level CQRS wiring layer in `src/wire.rs` that enforces query/view
registration via phantom-typed `Cons`/`Nil` lists. That wrapper exists because
`cqrs-es` itself accepts stringly-typed aggregate IDs and does not enforce that
query processors are wired to their aggregates.

We are switching to
[`event-sorcery`](https://github.com/ST0X-Technology/event-sorcery), which folds
the framework and the sqlite event store into a single crate we own. It will be
available on crates.io once published; until then we depend on it via a git
reference.

## Acceptance Criteria

- [ ] `Cargo.toml` replaces the `cqrs-es` dependency and the github `sqlite-es`
      reference with a single `github:ST0X-Technology/event-sorcery` git
      dependency (or a crates.io release once published).
- [ ] All existing aggregates (`Ingestion`, plus any others under `src/`)
      compile and pass their existing tests against `event-sorcery`.
- [ ] Aggregate IDs remain newtype-wrapped (no raw `String` at call sites); the
      `AggregateId` trait pattern from `src/wire.rs` is preserved or improved.
- [ ] Query/view wiring is enforced at compile time. Forgetting to register a
      view for an aggregate fails to compile, matching the guarantee the current
      `wire` module provides.
- [ ] The clippy `disallowed-methods` block on `sqlite_cqrs` is either kept (now
      pointing at the new equivalent) or removed because `event-sorcery` makes
      the unsafe path unreachable.
- [ ] `src/wire.rs` is either deleted or reduced to a thin shim with a one-
      paragraph docstring explaining what residual concern it still owns.
- [ ] No regression in observability: every event persistence and view update
      that currently emits a tracing log continues to emit an equivalent log at
      the same level.
- [ ] Existing migrations continue to apply cleanly; if `event-sorcery` requires
      schema changes, a forward-only sqlx migration is added.
- [ ] README or an ADR documents the choice of `event-sorcery` and the migration
      path so the next person reading `src/lib.rs` knows why the framework
      changed.
