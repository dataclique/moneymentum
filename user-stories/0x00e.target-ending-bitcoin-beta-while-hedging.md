# Target Ending Bitcoin Beta While Hedging

As a user, I want to choose my ending beta to Bitcoin while constructing a hedge
so that I can decide how much market exposure remains after rebalancing.

## Acceptance Criteria

- [ ] The portfolio page shows current Bitcoin beta and target Bitcoin beta.
- [ ] The user can enter a desired ending Bitcoin beta. Accepted input is a
      number in the range `-2.0` to `2.0` (negatives permitted for inverse
      exposure); blank, non-numeric, or out-of-range input is rejected with the
      inline message "Beta must be between -2.0 and 2.0". `0` is a valid target
      meaning fully hedged. `NaN` and empty inputs disable the staging action
      until corrected.
- [ ] The app previews the hedge gap between current beta and target beta. Hedge
      gap is defined as `hedge_gap = current_beta - target_beta` (beta points).
      Sign convention: a positive `hedge_gap` means the current portfolio beta
      is above the target -- i.e. the portfolio is **under-hedged** and needs
      additional hedging to bring beta down. A negative `hedge_gap` means the
      current beta is below the target -- i.e. the portfolio is **over-hedged**
      relative to the target. The UI preview text, the staging projection logic
      in the rebalancer, and any references to `hedge_gap`, `current_beta`, and
      `target_beta` use this single numeric convention. A dollar form
      `hedge_gap_dollars = hedge_gap * portfolio_notional` is shown next to the
      beta-points value and carries the same sign convention.
- [ ] Staged changes update the projected ending beta. The projection is an
      idealized calculation using current marks, position weights, and the same
      beta methodology as story 0x00b; it does not model slippage, fees, partial
      fills, or market impact, and is presented as an estimate, not a guarantee.
      The portfolio-projection logic in the rebalancer owns this computation and
      labels execution-realism gaps in the UI.
- [ ] If the requested target beta cannot be achieved with the current
      instrument universe and constraints, the portfolio page shows a clear
      "Cannot reach target with current universe/constraints" state, the
      projected ending beta preview marks the result as infeasible, the message
      names which constraint blocks the target (instrument coverage, liquidity,
      or maximum leverage), and the rebalance submit action is disabled until
      the target is achievable or revised.
- [ ] The app does not submit a rebalance solely because a target beta was
      entered; the user must still stage and submit trades.
