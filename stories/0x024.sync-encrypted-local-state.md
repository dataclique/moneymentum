# Sync Encrypted Local State Across Devices

As a user, I want my Hyperliquid keys, read-only wallet addresses, and other
local app state to persist across devices and sessions so that I do not have to
re-enter them every time I sign in on a new browser.

## Context

Hyperliquid orders are signed and submitted from the frontend, so the
Hyperliquid API key, the active account address, and similar per-user state must
live somewhere the frontend can read. Today that state lives in `localStorage`,
which is per-device and per-browser. A user who clears their storage, switches
browsers, or moves between machines has to re-enter everything by hand.

We persist that state server-side as an opaque encrypted blob. The blob is
encrypted client-side with a key derived from the user's connected Solana
wallet, so the server (and anyone with access to the server) cannot read it. The
server is only a dumb key-value store: `solana_pubkey -> encrypted_bytes`.

## Acceptance Criteria

- [ ] The backend exposes two endpoints keyed by Solana pubkey:
      `GET /portfolio-blob/{pubkey}` returns the most recent stored bytes (or
      404 if none), and `PUT /portfolio-blob/{pubkey}` stores new bytes. Both
      requests are authorized by a Solana-signature header proving control of
      `{pubkey}` over a short-lived server-issued nonce. The server never
      inspects, parses, or decrypts the body.
- [ ] The encryption scheme is documented and deterministic: the client derives
      a symmetric key from the user's Solana wallet by signing a fixed,
      protocol-defined message and hashing the signature, then encrypts the blob
      with an authenticated cipher (e.g. XChaCha20-Poly1305) using that key and
      a fresh random nonce stored with the ciphertext.
- [ ] The blob stores at minimum: the active Hyperliquid API key and address,
      the read-only wallets added in stories 0x00c and 0x012, and any
      preferences currently kept in `localStorage`. A versioned schema lives
      inside the encrypted plaintext so future fields can be added without
      breaking older clients.
- [ ] On wallet connect (after the Reown flow in story 0x00f) the client fetches
      the blob, decrypts it, and hydrates app state. If no blob exists the
      client treats the user as fresh and writes a new blob on the first state
      change.
- [ ] Whenever the relevant local state changes, the client writes a fresh blob
      to the backend (debounced) so the latest state is durable.
- [ ] If the server returns a blob but decryption fails (corrupt payload, wrong
      wallet, version mismatch) the client surfaces a clear user-facing error
      and does NOT silently overwrite the stored blob.
- [ ] The non-custodial security model holds: even with full backend compromise,
      the attacker cannot read the Hyperliquid API key or any other plaintext
      blob contents.
- [ ] A signed delete endpoint (or equivalent `PUT` of an empty payload) allows
      the user to clear their stored blob, for account reset or device hand-off.
